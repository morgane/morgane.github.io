Creating custom maps with Mapbox and Unity

a case study by Morgane Santos

ui design, ux design, prototyping, unity development

Mapbox creates a variety of tools to allow developers to build with live location data. The Unity team created an SDK letting people make their own interactive 3D maps. Over time, user research revealed that people wanted more and more customization power, and that they also found our UI confusing. We wanted to give users better ways to build exactly what they had in mind while also cleaning up our designs to make our SDK easier to learn and play with.

I led a small team of designers/developers to create the point of interest (POI) and building replacements features. Together we conducted user interviews, created mockups, prototyped and tested them, and eventually built them into the SDK. Afterwards I wrote tutorials and documentations for these new features. As team lead I was also responsible for communicating with other teams and making sure we were always on track.

Understanding the problem space

Since we had an existing SDK already, we decided to start with usability tests to see how people used it and where they got stuck. Between myself and 2 others, we ran tens of tests with existing customer.

A glimpse of all the usability test notes we took

This was extremely helpful. We were able to see where people were getting confused with our SDK and reassess our own biases. Time and time again, the issue of customization came up. Specifically, there were two common feature requests:

  1. Highlight points of interest (POIs) on the map
  2. Replace default buildings with custom meshes

Why POIs and buildings?

Before this project, the Mapbox Unity SDK still required people to do a fair amount of scripting to do any customization. It was possible to create a map with terrain and imagery (satellite imagery or map styles) with basic white rectangles extruded from building footprints to simulate real buildings. While this is fine for prototyping and proof-of-concepts, this quickly became a problem for anyone wanting to make a higher fidelity experience.

Specifically, people wanted to be able to highlight POIs for gameplay. Maybe you want to create an AR experience that triggers an event every time you're near a restaurant. Maybe you want to make a 3D map that labels specific museums in downtown San Franciso.

As for buildings, people were rightfully tired of the basic white rectangles. It was incredibly difficult to style these rectangles, let alone change the meshes altogether. We specifically tackled mesh replacement but others on the team also focused on making it easy to add custom materials to buildings.

Without POI highlighting or building replacement, the Mapbox Unity SDK was only useful for quick prototyping. By focusing on these user needs, the Mapbox Unity SDK would allow people to create their 3D maps from start to finish in one place.

Early flows and mockups

Before really diving into the design, I created a Unity UI kit for Figma so we could quickly iterate over designs (rather than having to create a dropdown menu over and over again).

A Unity UI kit helped us iterate through designs much faster

The two main issues (highlighting POIs and replacing buildings) had surprisingly similar flows. In both cases, users may want to highlight/replace a specific building (the Empire State Building) or make procedural changes to a category of buildings (all museums in a city). While the UI would have to look different, since building replacement had more requirements, the flows had similar patterns.

Illustrations to help communicate and align on what features were required

There were a few specific feature requirements. People needed to be able to select POIs and buildings by category, name, or lat-lon location. They'd need to be able to add custom assets and meshes in both cases and potentially make the POIs or buildings interactive. And this would all need to make sense to both mapping experts and people completely new to Unity and Mapbox.

We worked through user flows and marked what assumptions we'd need to test at each step. This helped guide the UI design for our SDK. We wanted robust UI to allow people to do all the customizations they wanted with minimal coding. This also meant the UI had to be well-organized and easy to follow; previous SDK versions were unwieldly for people and we wanted to improve on this.

Bringing it into the SDK

We turned these mockups into clickable prototypes and tested them with in-person interviews. After refining our designs, we were ready to add them to the SDK. The backend engineers had been working on making these features possible, so we just had to code the UI to expose it to users. Once this was done, the whole team tested these features extensively to find bugs or areas we overlooked.

In conclusion

These features made a huge impact for our users. By starting with so much user research, we were able to design for real needs and not what we assumed was needed. A heavy emphasis on iterative design allowed us to refine the complex UI into something that was understandable even to first-time Unity or Mapbox users. Whereas previously people needed a solid understanding of mapping terms, now anyone could jump in quickly without needing to be a mapping expert.
